🚨 问题：内网服务直接暴露到公网

以 SSH 服务为例，将内网 SSH 端口映射到公网后：

⚠️ SSH 暴力破解攻击

互联网上存在大量自动化扫描工具，持续对公网 IP 的 22 端口进行密码暴力破解。一个暴露在公网的 SSH 服务，每分钟可能遭受数十次甚至上百次登录尝试。

🔓 安全风险

  • 弱密码被破解 → 服务器被入侵
  • 即使使用密钥认证，暴力破解仍会产生大量无效连接
  • 攻击者可能利用 SSH 漏洞进行远程代码执行
  • 服务器成为跳板，被用于攻击其他目标

💥 性能压力

  • 大量无效连接消耗 CPU 和内存资源
  • SSH 守护进程需要处理每次认证请求
  • 日志文件快速膨胀，影响磁盘空间
  • 正常用户的连接质量受到影响
# 典型的 SSH 暴力破解日志

Jun 14 08:23:02 server sshd[12346]: Failed password for admin from 5.6.7.8 port 12345
Jun 14 08:23:03 server sshd[12347]: Failed password for root from 9.10.11.12 port 67890
... 每分钟数十条类似记录

🔐 解决方案：STCP（Secret TCP）

核心思想：不再将服务端口直接暴露到公网，而是通过一个共享密钥（Secret Key）来控制访问权限。只有持有正确密钥的访客才能连接。

服务不监听公网端口

⚙️ STCP 工作原理

STCP 涉及三个角色：服务端 xfrpc、frps 服务器、访客端 xfrpc

� 用户
127.0.0.1:6000
→
💻 AW (访客端)
xfrpc stcp_visitor
→
☁️ openclaw (中转)
frps :7070
→
📡 Nokia (服务端)
xfrpc stcp · SSH :22

服务端 xfrpc 配置（Nokia）

# /etc/config/xfrpc
config xfrpc 'common'
    option server_addr '43.159.38.154'
    option server_port '7070'
    option token '...'
    option tls_enable '1'
    option tls_cert_file '/etc/xfrpc-client.crt'
    option tls_key_file '/etc/xfrpc-client.key'
    option tls_trusted_ca_file '/etc/xfrpc-ca.crt'

config stcp 'nokia_ssh'
    option local_ip '127.0.0.1'
    option local_port '22'
    option secret_key 'stcp_nokia_ssh_2026'

Nokia 通过 OpenWrt UCI 注册 STCP 代理，指定密钥（secret_key）。

访客端 xfrpc 配置（AW）

# /etc/config/xfrpc
config xfrpc 'common'
    option server_addr '43.159.38.154'
    option server_port '7070'
    option token '...'
    option tls_enable '1'
    option tls_cert_file '/etc/xfrpc-client.crt'
    option tls_key_file '/etc/xfrpc-client.key'
    option tls_trusted_ca_file '/etc/xfrpc-ca.crt'

config stcp_visitor 'nokia_ssh_visitor'
    option server_name 'nokia_ssh'
    option bind_addr '127.0.0.1'
    option bind_port '6000'
    option secret_key 'stcp_nokia_ssh_2026'

AW 通过 OpenWrt UCI 注册 STCP 访客，指定密钥和本地监听端口。

⚠️ 常见配置错误：

  • UCI 中密钥字段为 secret_key，init 脚本会自动转换为 INI 的 sk 字段
  • 缺少 TLS 证书配置 → 连接被 frps 拒绝（transport.tls.force = true）
  • 忘记设置 loglevel 选项 → xfrpc 因 -d 参数为空而崩溃

关键流程：当远程用户连接访客端的 6000 端口时，访客端 xfrpc 通过 frps 向服务端发起连接请求（携带密钥），frps 验证密钥匹配后建立数据通道，实现端到端通信。

🛡️ STCP 如何解决安全问题

❌ 普通 TCP 代理场景

# frps 监听公网端口 6000
# 任何人都可以尝试连接
$ nmap -p 6000 公网IP
PORT     STATE SERVICE
6000/tcp open  X11

# 攻击者可以暴力破解
$ hydra -l root -P wordlist.txt ssh://公网IP:6000

❌ 端口对所有人可见，攻击面大

✅ STCP 代理场景

# frps 不在公网暴露 SSH 端口
# 攻击者扫描不到任何开放端口
$ nmap -p 6000 公网IP
PORT     STATE SERVICE
6000/tcp closed X11

# 即使知道端口，没有密钥也无法连接
# frps 内部验证密钥后才建立通道

✅ 无公网端口暴露，零攻击面

双重安全保障

  • 第一层（STCP 密钥）：frps 在转发连接前验证密钥，密钥不匹配则直接拒绝
  • 第二层（SSH 认证）：即使密钥验证通过，用户仍需通过 SSH 密钥/密码认证
  • 零暴露面：内网服务端口不在公网监听，扫描工具无法发现

📝 配置详解

☁️ frps 服务器配置 (nwct-server.toml)

# /etc/nwct/nwct-server.toml
bindPort = 7070
auth.method = "token"
auth.token = "fc21625021dd5517c3de0d939bd1ede627ee53a1af4eaaa5"

# TLS 强制模式（必须配置客户端证书）
transport.tls.force = true
transport.tls.certFile = "/etc/nwct/certs/server.crt"
transport.tls.keyFile = "/etc/nwct/certs/server.key"
transport.tls.trustedCaFile = "/etc/nwct/certs/ca.crt"

# 可选：QUIC 支持
# quicBindPort = 6001

关键配置说明：

  • server_name 必须与服务端代理的 section 名称一致（如 nokia_ssh）
  • secret_key 两端必须完全相同，init 脚本会自动映射为 xfrpc 的 sk 字段
  • TLS 证书由 frps 的 CA 签发，客户端需安装 CA 证书 + 客户端证书/密钥

📜 TLS 证书生成流程

frps 启用了 transport.tls.force = true，需要为服务器和客户端分别生成证书，均由同一个 CA 签发。

Step 1: 生成 CA 根证书

# 生成 CA 私钥
openssl genrsa -out ca.key 2048

# 生成 CA 自签名证书（有效期 10 年）
openssl req -new -x509 -days 3650 -key ca.key -out ca.crt \
  -subj "/CN=xfrpc-ca"

Step 2: 生成 frps 服务器证书

# 生成服务器私钥
openssl genrsa -out server.key 2048

# 生成证书签名请求（CN 填服务器 IP 或域名）
openssl req -new -key server.key -out server.csr \
  -subj "/CN=43.159.38.154"

# 用 CA 签发服务器证书（有效期 10 年）
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -out server.crt -days 3650

Step 3: 生成客户端证书

# 生成客户端私钥
openssl genrsa -out client.key 2048

# 生成证书签名请求
openssl req -new -key client.key -out client.csr \
  -subj "/CN=xfrpc-client"

# 用 CA 签发客户端证书（有效期 10 年）
openssl x509 -req -in client.csr -CA ca.crt -CAkey ca.key \
  -CAcreateserial -out client.crt -days 3650

生成示例（一键脚本）

#!/bin/bash
# 一键生成 CA、服务器、客户端证书
set -e

# CA 根证书
openssl genrsa -out ca.key 2048
openssl req -new -x509 -days 3650 -key ca.key -out ca.crt -subj "/CN=xfrpc-ca"

# 服务器证书（CN 填 frps 服务器 IP 或域名）
openssl genrsa -out server.key 2048
openssl req -new -key server.key -out server.csr -subj "/CN=43.159.38.154"
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out server.crt -days 3650

# 客户端证书（所有 xfrpc 客户端共用）
openssl genrsa -out client.key 2048
openssl req -new -key client.key -out client.csr -subj "/CN=xfrpc-client"
openssl x509 -req -in client.csr -CA ca.crt -CAkey ca.key -CAcreateserial -out client.crt -days 3650

# 清理 CSR 和序列文件
rm -f *.csr *.srl
echo "Done: ca.crt, server.crt, client.crt"

⚠️ 注意事项：

  • CA 私钥（ca.key）必须妥善保管，泄露后可伪造任意客户端证书
  • 客户端证书必须由同一个 CA 签发，否则 frps 的 trustedCaFile 验证会失败
  • 私钥文件权限应设为 600，仅 owner 可读写

🚀 快速验证步骤

Step 1: 确认 frps 运行

# openclaw VPS
sudo systemctl status nwct-server
sudo ss -tlnp | grep 7070

nwct-server (frps) 已通过 systemd 管理

Step 2: 确认 Nokia 服务端运行

# Nokia (通过 openclaw 中转)
ssh nokia "/etc/init.d/xfrpc status"
# 确认 STCP 代理已注册
ssh nokia "logread -e xfrpc | grep nokia_ssh"

xfrpc 服务端需先注册 STCP 代理

Step 3: 确认 AW 访客端运行

# AW 设备
ssh aw "/etc/init.d/xfrpc status"
ssh aw "netstat -tlnp | grep 6000"

确认 6000 端口已监听

Step 4: 测试 STCP 隧道

# 在 AW 上通过 STCP 隧道 SSH 连接到 Nokia
ssh -p 6000 root@127.0.0.1

# 验证连接成功 - 实际连接的是 Nokia（服务端）
root@chawrt:~# uptime
 21:09:06 up 13 days, load average: 0.00, 0.01, 0.00

# 验证安全性 - frps 公网端口无 STCP 端口暴露
nmap -p 6000 43.159.38.154
# 结果: 6000/tcp closed  ← 端口未暴露！

✅ 验证成功标志：SSH 连接正常建立（实际连到 Nokia），且 frps 公网上无对应端口暴露。流量通过 TLS 加密隧道传输。

调试技巧：

  • 使用 -d 7 参数开启 xfrpc 调试日志
  • OpenWrt 上使用 logread -e xfrpc 查看日志
  • 检查 frps 日志：sudo journalctl -u nwct-server -f
  • Nokia 需通过 openclaw 中转访问：ssh -J openclaw root@192.168.9.1
