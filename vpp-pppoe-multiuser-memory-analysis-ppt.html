项目背景

  • 使用 VPP 实现 PPPoE 多用户拨号
  • 面向万级用户规模
  • 需要用户三层完全隔离
  • 技术方案：每个拨号用户独立 VRF

问题现象

  • 小规模用户运行正常，无内存异常
  • 随着用户数量增加，内存快速增长，多次触发 VPP 崩溃
  • 增加 VPP 内存配置后问题有所缓解，但不彻底
  • 2 万用户测试中内存占用达到不可接受水平

排查思路

  • 排除数据面 buffer 泄漏
  • 排除 PPPoE session 状态异常
  • 发现内存增长与 VRF 数量强相关
  • 判断问题在 VRF 相关结构

VRF 与 FIB 关系

  • VPP 中每个 VRF 对应一套路由表（FIB）
  • FIB 内部维护完整路由查询结构
  • PPPoE 场景中：用户数 ≈ VRF 数量
  • 需要维护大量 FIB 实例
工程项目实战分享

VPP 默认路由算法

  • IPv4 FIB 默认采用 16-8-8 查询结构
  • IPv4 地址拆分为：
  • 前 16 bit
  • 中间 8 bit
  • 最后 8 bit
  • 设计目标：极致查询性能
工程项目实战分享

16-8-8 内存模型

typedef struct ip4_mtrie_16_ply_t_
{
  ip4_mtrie_leaf_t leaves[65536];        // 65536 * 4 = 262144 bytes
  u8 dst_address_bits_of_leaves[65536];  // 65536 * 1 = 65536 bytes
} ip4_mtrie_16_ply_t;                    // Total: ~320KB
  • 一级表大小：2^16 个 entry
  • 每个 entry 占用 5 字节
  • 单 VRF 固定占用约 320 KB
  • 即使只有一条默认路由也无法减少

规模放大后的结果

  • VRF 数量：10000
  • 内存占用：320 KB × 10000 ≈ 3.2 GB
  • 内存消耗与实际路由数量无关
  • 成为主要内存瓶颈

16-8-8 的设计初衷

  • 查询层级少
  • 内存访问路径稳定
  • cache 命中率高
  • 适用于少 VRF、大路由场景

问题本质总结

  • 不是内存泄漏
  • 是算法设计与使用场景不匹配
  • 海量 VRF + 小路由表
  • 固定开销被放大

优化方向

  • 目标：降低每个 VRF 的固定内存成本
  • 场景特点：VRF 多、路由少
  • 引出方案：8-8-8-8 路由查询结构
工程项目实战分享

8-8-8-8 路由结构

  • IPv4 拆分为 4 个 8 bit
  • 每层最多 256 个 entry
  • 路由表按需分配
  • 无大块预分配表
工程项目实战分享

两种算法对比

  • 内存占用：
  • 16-8-8：固定320KB一级表开销
  • 8-8-8-8：按需分配，随路由增长
  • 查询性能：
  • 16-8-8：更快
  • 8-8-8-8：略慢
  • 适用场景明显不同

方案选择

  • PPPoE 多用户特点：
  • VRF 极多
  • 路由极少
  • 性能不是瓶颈
  • 内存是核心限制
  • 选择更省内存方案

VPP 8-8-8-8 启用方法

  • VPP 默认使用 16-8-8 算法
  • 要启用 8-8-8-8，修改 CMake 选项
  • 在 src/vnet/CMakeLists.txt 中设置：
  • option(VPP_IP_FIB_MTRIE_16 OFF)
  • 或构建时添加 -DVPP_IP_FIB_MTRIE_16=OFF
  • 重新编译 VPP 并部署
