基于 VPP 的 PPPoE 多用户拨号
内存问题深度分析

VRF 隔离场景下的路由表设计取舍

[ 工程项目实战分享 · staylight ]

项目背景

  • 使用 VPP 实现 万级用户 PPPoE 拨号
  • 核心需求：用户之间实现 三层完全隔离
  • 方案：分配独立 VRF (Virtual Routing & Forwarding)
  • 挑战：VRF 规模与系统资源的博弈

典型“内存爆炸”现场

  • 100 用户量：内存占用平稳，响应迅速
  • 5,000 用户量：内存线性增长，开始触发抖动
  • 15,000 用户量：VPP 内存碎片溢出，进程崩溃
  • 直观感受：每个 VRF 似乎都在吞噬固定份额的内存

VPP FIB 与 16-8-8 结构

IPv4 FIB 默认采用 16-8-8 的 Mtrie 查询结构：

  • 一级表（Ply 16）固定预分配 2^16 = 65536 个条目
  • 每个单 VRF 固定开销：~320 KB
  • 无论内部有 1 条路由还是 10,000 条，开销不变

规模放大的后果

10,000 VRF x 320 KB/VRF = 3.2 GB
  • 这仅仅是“底色”内存开销，未计算其他控制结构
  • 对于内存受限的边缘设备，这是灾难性的
  • 原因：极致查询性能与内存占用的空间换时间博弈

更均衡的方案：8-8-8-8

将查询层级变深，每一层的 Ply 变得更小：

  • Ply 8 仅需 256 个条目
  • 单 VRF 初始开销降至 ~1.3 KB
  • 牺牲了 1-2 次内存访问，但极大地换回了空间

实测对比：万级路由表创建

seq 10000 | xargs -I {} vppctl ip table add {}

16-8-8 (默认)

4.92 GB

内存压力极大，扩容成本高

8-8-8-8 (优化)

1.28 GB

内存下降 ~74%

如何启用部署？

在编译阶段通过 CMake 参数控制：

# 修改 vnet/CMakeLists.txt 
option(VPP_IP_FIB_MTRIE_16 "Use 16-8-8 FIB" OFF)

# 或者命令行构建
cmake .. -DVPP_IP_FIB_MTRIE_16=OFF
  • 本质上是根据业务场景重新选择算法参数
  • 对于万级用户 PPPoE，这几乎是必选项

总结与建议

  • 16-8-8：适合核心网，VRF 少、路由表大且追求极致线速
  • 8-8-8-8：适合接入网/CPE，VRF 极多、路由表小且内存敏感
  • 技术没有绝对优劣，只有 场景适配
