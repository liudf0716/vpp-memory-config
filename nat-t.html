为什么 IPsec VPN 无法穿透 NAT？
NAT-T 机制深度解析

NAT Traversal: From Conflict to Resolution

RFC 3947 / 3948 · 技术专题解析

IPsec 与 NAT 的“水火不容”

  • IPsec 依赖端到端的 IP 地址、端口不变性
  • NAT 会修改三层 IP 地址与四层端口信息
  • 这破坏了 IPsec 的 安全假设 与数据完整性校验
  • NAT-T 机制：在工程妥协中寻求通信方案

什么是 NAT-T？

  • NAT-T（NAT Traversal）是 IPsec 的传输扩展机制
  • 核心使命：解决 IPsec 在 NAT 网络中的 不可达 问题
  • 标准规范：定义于 RFC 3947 / RFC 3948

原生 IPsec 的 NAT 困境

  • ESP 使用协议号 50，没有 L4 端口（TCP/UDP）
  • NAT 无法基于“端口”为 ESP 建立映射状态表
  • 多个内网用户拨号时，NAT 无法区分回程流量

NAT-T 的核心：换个马甲

  • 放弃 原生 ESP 直接传输
  • 将 ESP 数据包封装进 UDP 头部
  • 让 NAT 设备把 IPsec 当作普通 UDP 流量处理

为什么是 UDP 4500？

  • 避免与 IKE 初始协商端口 500 冲突
  • RFC 3948 明确定义的标准“隧道端口”
  • 便于企业防火墙进行统一的安全策略配置

必须进行的“先置动作”

  • NAT-T 并非总是开启，因为它会带来 8 字节 封装开销
  • 通信双方必须先探测是否存在 NAT 设备
  • 检测机制：NAT Detection (NAT-D)

NAT-D 的数学原理

IKE_SA_INIT 阶段，双方计算地址指纹：

Hash = SHA1(SPI-I | SPI-R | IP | Port)
  • 交换 Hash 后，若本地重算结果不同 → 发现 NAT
  • 这是一种不可篡改的“地址指纹”校验

检测到 NAT 之后的行为

  • 端口浮动 (Port Floating)：协商从 500 切换至 4500
  • 统一通道：所有控制流与数据流在一条 UDP 连接复用
  • 解决校验：IKE 报文不再 because IP 漂移而验证失败

双流量复用：如何区分？

UDP 4500 同时走 IKE 和 ESP，网卡如何分拣？

  • 利用报文前 4 字节的 Demultiplexing 标记
  • IKE 包：头部插入 4 字节全零 (Non-ESP Marker)
  • ESP 包：SPI 字段天然不为零，直接区分

关键优势总结

  • 生存能力：穿透限制，实现 VPN 连接
  • 性能平衡：仅引入极少的封装字节头
  • 广泛兼容：几乎所有现代 IPsec 实现的标配

结论

NAT-T 不是安全增强，而是生存手段。

  • 它让安全性极高的 IPsec 能在充满 NAT 的互联网中存活
  • NAT-D 是开启这扇大门的关键钥匙
